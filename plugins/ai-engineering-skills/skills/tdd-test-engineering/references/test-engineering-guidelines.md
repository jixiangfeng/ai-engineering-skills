# 测试工程指南

本指南约束 `tdd-test-engineering` 的执行纪律。目标不是生成测试代码片段，而是形成可追踪的测试闭环。

## 核心流程

1. 读取需求、设计、接口、代码、Git Diff、已有测试和上游 handoff。
2. 将需求转换成验收标准。
3. 分析直接和间接影响范围。
4. 设计测试策略和测试用例。
5. 等待测试用例确认。
6. 验证测试环境、测试账号、测试数据和清理方案。
7. 编写或补充自动化测试。
8. 执行 Red / Green / Regression 相关命令。
9. 记录问题、失败分类、执行证据和剩余风险。
10. 输出测试总结或下游 handoff。

## 测试用例确认

测试用例确认前，不允许：
- 大范围新增测试。
- 修改生产业务代码。
- 修改业务预期。
- 开始环境有写入风险的集成测试。

测试用例至少记录：
- case id
- 关联验收标准或问题编号
- 优先级
- 测试层级
- 前置条件
- 输入和步骤
- 预期结果
- 是否自动化
- 数据清理要求

## TDD 执行

新功能可使用 `Red -> Green -> Refactor`：
- Red：先写测试，并确认失败原因是功能尚未实现或缺陷仍存在。
- Green：只做让测试通过的最小实现。
- Refactor：测试通过后才允许整理结构，每次整理后重新执行相关测试。

已有功能和 bug 修复不强行套用新功能 TDD，应先识别当前行为，再补充特征测试或缺陷复现测试。

## 环境和凭证

集成测试前必须先过环境安全门，再确认环境类型和写入边界。禁止自动写生产库、清空共享库、清空 Redis、发送真实短信、调用真实支付、触发生产 MQ。

## 环境安全门

第一步必须执行环境安全门，检查：
- 目标环境是否明确是 local / test / staging，而不是 production
- 连接目标是否都来自环境 profile
- 每个服务是否明确写出允许操作和禁止操作
- 是否禁止危险写入，直到用户对当前私有环境做了单次授权

环境安全门未通过时，不得开始连接 MySQL、MongoDB、Redis、MQ、Nacos、Elasticsearch、外部 API 或模型服务。

默认记录：
- 主机、端口、库名、用户名
- 测试环境标识
- 凭证来源
- 脱敏值
- 连通状态

如果用户明确说明是个人本地使用，并允许明文记录，可以把测试环境密码、Token 或 API Key 写入环境 profile 或本轮测试文档。此时必须记录：
- `credentialPolicy: plaintext_user_approved`
- 用户批准依据
- 凭证适用环境
- 是否允许 handoff 携带凭证
- 风险说明

生产凭证、私钥、真实敏感用户数据仍不应默认落文档；除非用户明确要求写入私人本地文档。

推荐把固定环境配置沉淀到：

```text
workflow/env-profiles/<env>.yaml
```

本轮测试文档只引用 profile，并记录本轮 override 和验证结果。

环境验证按四层记录：
- 网络连通：DNS、TCP、TLS、代理、防火墙和超时。
- 认证验证：账号、Token、目标库/命名空间权限、只读或测试写权限。
- 服务验证：MySQL `SELECT 1`、Redis `PING`、Neo4j `RETURN 1`、Nacos 只读配置查询、外部接口健康检查。
- 业务数据验证：表结构、版本、字典、测试账号、测试数据和清理脚本。

P0 用例依赖的环境未通过时，不得开始完整测试。

## 失败分类

每个失败必须归类：
- 业务代码缺陷
- 测试用例错误
- 测试数据问题
- 环境问题
- 外部依赖问题
- 历史测试已失败
- 非确定性偶发失败
- 配置错误
- 需求预期不明确

## 问题确认与修复门

发现生产代码疑似缺陷时，先输出问题证据，再等待用户确认修复范围。问题记录至少包含：
- 问题编号、严重级别和是否阻塞发布
- 关联测试用例和验收标准
- 测试环境、前置条件、复现步骤、预期结果、实际结果
- 执行命令、日志或报告路径
- 初步原因、建议修复范围和修复风险

用户未确认前，不修改生产业务代码。用户确认后，先确认失败测试可稳定复现，再做最小修复；不得通过放宽断言、删除测试、`@Disabled`、吞异常或改业务预期来让测试变绿。

## 回归阶梯

修复后按风险执行回归：
1. 缺陷用例回归：原失败用例必须从失败变为通过。
2. 相邻场景回归：覆盖边界、空值、权限变化、外部依赖异常、重复请求和历史 Bug。
3. 模块回归：执行当前业务模块测试。
4. 完整回归：执行所有已确认测试用例。

高风险场景增加并发、性能、数据一致性、外部契约和清理验证。

## 完成条件

只有同时满足以下条件，才能声明测试完成：
- 已批准测试用例都已执行或有确认过的阻塞记录。
- P0 / P1 用例未被无说明跳过。
- 没有未解释失败。
- 环境和测试数据状态明确。
- 测试数据清理完成或清理风险已记录。
- 执行命令、结果、报告路径和关键证据已记录。
- 生产代码变更后完成缺陷回归和必要模块回归。

## AI 功能评测

涉及模型、prompt、智能问诊、排序或抽取时，普通断言不足。应记录：
- 数据集版本
- 期望输出版本
- 模型和 prompt 版本
- 温度、重复次数和成本
- JSON 合规率、准确率、召回率、TopK、MRR、NDCG、危险项漏召回等指标
- 与基线对比和阈值
