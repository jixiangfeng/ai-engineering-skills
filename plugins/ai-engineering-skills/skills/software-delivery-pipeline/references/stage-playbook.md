# 阶段执行手册

本文说明如何按纪律执行每个阶段。所有阶段产物默认使用简体中文，代码标识符、命令、路径、错误输出、API 名称和用户原文可以保留原文。

## 澄清与收敛循环

- 需求、架构、计划三个确认门禁都是循环，不是一轮确认。
- 不把用户原话当成天然正确；必须结合代码证据、项目约束、风险和可行性判断。
- 发现矛盾、缺口、不可行、高风险或与代码现状冲突时，明确指出，提出选项，更新当前阶段文档，再请求确认。
- 用户反馈不能只停留在聊天里，必须写回当前阶段文档。
- 只有文档状态已确认且无未解决阻塞项，才能进入下一阶段。

## 需求确认

- 将模糊请求收敛成可验证的工程目标。
- 明确范围内、范围外、约束、假设和验收标准。
- 写完 `01-delivery-requirements.md` 后必须暂停，向用户展示运行目录、需求摘要和需要确认的问题。
- 用户提出补充或修改时，只更新 `01-delivery-requirements.md`，再次请求确认，不进入计划或实现。
- 只有用户明确确认需求后，才允许进入计划文档：简单任务使用 `02-delivery-plan.md`，架构门禁任务使用 `03-delivery-plan.md`。
- 如果缺少关键信息会影响安全实现，先问一个简短问题，并把该问题记录到需求文档。

## 架构设计与技术选型

- 默认保持轻量：小修复不需要单独架构文档，只在计划中说明无需独立架构设计。
- 触发条件包括跨模块/跨服务、持久化结构、外部依赖、API/数据契约、异步流程、权限安全、数据一致性、review 标记为架构级问题，或用户明确要求设计/架构/选型/方案。
- 触发时写 `02-delivery-architecture.md`，比较方案、说明取舍、定义边界和兼容策略。
- 用户确认架构前，不允许进入实施计划或代码修改。

## 计划

- 识别受影响的代码、接口和验证检查。
- 计划要短、按顺序排列，并提前考虑测试。
- 好的计划应当让另一个 agent 能直接据此实施。
- 写完当前计划文档后必须暂停，向用户展示计划摘要、关键风险和验证策略；简单任务使用 `02-delivery-plan.md`，架构门禁任务使用 `03-delivery-plan.md`。
- 用户确认计划前，不允许进入实现或修改代码。
- 用户要求调整时，只更新当前计划文档并再次请求确认。

## 测试优先实施

- 优先在改代码前写能失败的测试。
- 使用最小但有意义的测试切片。
- 如果无法先写测试，说明原因。
- 只做让验证通过所需的最小实现，然后重新运行检查。
- 如果需要扩大范围或明显偏离已确认计划，先记录原因并暂停请求用户确认。

## 系统化调试

- 不从猜测性修复开始。
- 记录准确错误。
- 尽量稳定复现。
- 检查近期改动和配置差异。
- 沿数据流和控制流向源头回溯。
- 修根因，不只修表象。
- 如果根因修复会改变已确认范围、架构、持久化、兼容策略或需要高风险操作，先暂停请求用户确认。

## 验证

- 使用最小但有证明力的验证门禁。
- 优先使用项目原生命令，而不是临时发明检查。
- 如果完整套件成本太高，运行最接近的窄门禁并说明原因。
- 当存在可运行验证时，单独 diff review 不足以证明完成。
- 如果关键验证被环境、权限、数据或外部依赖阻塞，记录阻塞并暂停，让用户决定是否接受风险或补齐条件。

## 交付

- 交付是摘要，不是流水账。
- 说明修改文件、验证证据和剩余风险。
- 保持足够短，方便人工快速扫描。
- 输出最终交付摘要后等待用户验收或反馈；简单低风险路径默认使用 `06-delivery-summary.md`，复杂路径使用 `08-delivery-summary.md`。如用户不接受，按反馈回到对应阶段更新文档和实现。
