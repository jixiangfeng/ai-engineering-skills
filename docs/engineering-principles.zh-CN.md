# Engineering Principles

本文定义 `ai-engineering-skills` 的跨 workflow 工程行为底线。它不是新的 workflow，也不替代各 skill 的阶段规则；它用于约束所有 workflow 在理解、设计、实现、验证时的默认行为。

## 1. 编码前先澄清

- 不把用户的初始描述自动当成完整需求。
- 如果存在多种解释，先写明解释差异、影响和推荐路径。
- 如果关键事实无法从代码、配置、测试或文档确认，标记为 `待确认`。
- 对高风险或不可逆操作，必须先暂停确认。

## 2. 简单优先

- 只解决当前确认的范围，不添加未要求的能力。
- 不为单次使用场景抽象接口、策略、框架或配置层。
- 优先沿用项目已有结构和命名，不发明新的术语体系。
- 如果一个方案明显比另一个方案更小、更直接，应说明取舍。

## 3. 精准修改

- 每个改动都必须能追溯到需求、finding、root cause、contract 或 migration plan。
- 不做顺手格式化、换行归一化、无关重构或“顺便清理”。
- 发现无关问题时，记录为后续 review 线索，不直接修。
- 对已有本地改动，默认视为用户改动；不得覆盖或回滚。

## 4. 目标驱动验证

- 需求、finding、root cause、contract、migration plan 都应对应可验证结果。
- 完成声明必须基于已经执行的命令、测试、手工检查，或明确说明验证阻塞原因。
- 验证失败不能进入完成总结，只能进入调试、返工或阻塞说明。
- 如果无法运行推荐验证，必须记录原因、剩余风险和替代检查。

## 5. 文档优先但不过度文档化

- workflow 的关键结论必须写入阶段文档，不能只停留在聊天里。
- 简单任务可以使用简化路径，但仍要保留范围、验证和结果。
- 文档应服务于恢复、review 和 handoff；避免空泛叙述。

## 使用方式

- `workflow-bootstrap` 负责在路由前应用这些原则，尤其是澄清和范围判断。
- `codebase-orientation`、`code-review-triage`、`debug-root-cause`、`api-contract-design`、`data-migration-planning` 默认只读，使用这些原则控制证据、范围和待确认项。
- `software-delivery-pipeline` 在需求、计划、实现、验证和总结阶段都必须遵守这些原则。
