# 需求澄清模块

本模块用于所有开发、修改、重构、设计、排查类请求的入口判断。目标是在进入具体 workflow 前，把目标、范围和验收信号讲清楚，同时避免对小任务过度提问。

## 清晰度判断

处理请求前先判断：

- 目标是否明确。
- 修改或分析范围是否明确。
- 非目标是否明确。
- 验收标准是否明确。
- 是否存在业务歧义。
- 是否存在技术边界不清。
- 是否需要先熟悉代码、review、debug、契约设计或迁移规划。

## 任务大小

小任务：

- 可以直接做合理假设。
- 必须显式说明关键假设。
- 不要提出超过必要的问题。

中等任务：

- 最多询问 1-3 个关键问题。
- 问题必须直接关联范围、目标、验收标准或风险边界。
- 如果可以用代码证据确认，不要把问题丢给用户。

大任务：

- 先进入澄清阶段。
- 明确目标、非目标、输入范围、修改范围、验收标准、风险边界。
- 再选择具体 workflow。

## 输出结构

```md
## 需求澄清

### 已明确
- ...

### 仍不明确
- ...

### 本次默认假设
- ...

### 推荐进入的 workflow
- ...
```

## 停止条件

如果缺失的信息会改变 workflow 路由、修改范围、数据/接口行为、验证方式或回滚路径，必须暂停确认。不能用含糊假设继续。

## 正例

- 用户说“修复这个接口偶发 500”：先判断是 `debug-root-cause`，说明需要错误日志、复现步骤或失败请求样例。
- 用户说“给订单详情加字段”：先确认字段位置、调用方、兼容性和验收样例，再进入 `api-contract-design`。

## 反例

- 用户只说“优化下系统”就直接改代码。
- 发现 scope 不清却用“我理解应该是...”继续写实现。
