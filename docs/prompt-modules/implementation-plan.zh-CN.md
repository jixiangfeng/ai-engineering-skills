# Implementation Plan 模块

本模块用于所有代码修改类任务。没有实现计划，不允许进入代码修改；如果用户明确要求“直接改”，也要先给出极简计划。

## 必答问题

实现计划必须回答：

- 改哪里。
- 为什么改。
- 怎么改。
- 风险是什么。
- 怎么验证。
- 怎么回滚。

## 输出结构

```md
## Implementation Plan

### 1. 修改目标
- ...

### 2. 修改范围
- 涉及模块：
- 涉及文件：
- 不修改范围：

### 3. 执行步骤
1. ...
2. ...
3. ...

### 4. 数据 / 配置影响
- 数据库：
- Redis：
- MQ：
- 配置项：
- 外部接口：

### 5. 风险点
- ...

### 6. 验证方式
- 单元测试：
- 集成测试：
- 手工验证：
- 回归范围：

### 7. 回滚方式
- ...
```

## 强规则

- 计划必须先于代码修改。
- 计划必须锁定 scope。
- 如果计划不再适用，先修订计划并说明原因，再继续。
- 计划不能包含未确认的扩展需求。

## 正例

- 计划明确只改 `OrderService` 和 `OrderServiceTest`，并说明不改支付回调。
- 计划列出验证命令和回滚方式，再进入实现。

## 反例

- “我会看情况改相关文件”。
- 没有验证方式，只写“改完测试一下”。
