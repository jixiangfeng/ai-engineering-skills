# Review 准则

## Finding 质量标准

每个 finding 必须回答：
- 哪里有问题：文件路径和行号，或明确调用链/配置路径。
- 为什么是问题：用代码证据说明会导致什么错误、风险或维护成本。
- 影响是什么：用户行为、数据一致性、安全、性能、可维护性或测试可靠性。
- 怎么修：给方向，不在 review 阶段直接实现。
- 置信度：高、中、低；低置信度问题要标明待确认。

## 严重级别

- 阻塞：会导致核心流程不可用、数据损坏、安全漏洞、构建失败，或上线必须阻止。
- 高：明确的运行时 bug、接口契约破坏、重要边界缺失、明显回归风险。
- 中：局部错误、可触发但非核心路径的问题、重要测试缺口、维护风险较高。
- 低：小范围可维护性问题、边界体验问题、非关键测试或文档缺口。

## 不应作为必须修复的问题

- 只有风格偏好，没有实际风险。
- 未被当前请求覆盖的泛化重构。
- 与现有项目风格冲突的“最佳实践”。
- 无证据的性能、安全或并发猜测。

## Review 输出顺序

1. Findings 按严重级别排序。
2. 每个 finding 用稳定 ID：`F-001`、`F-002`。
3. 先列问题，再列开放问题和剩余风险。
4. 如果没有问题，明确写“未发现需要处理的问题”，并列出审查范围和未覆盖验证。

## 修复阶段规则

- 用户确认修复项前不改代码。
- 用户确认修复计划前不改代码。
- 只修用户选择的 finding。
- 发现新问题时追加记录并暂停确认，不偷偷扩大范围。

## 交接到 software-delivery-pipeline

当用户确认修复计划并希望继续修复时，生成 `review-to-delivery-handoff.md`。该文件是 review skill 和 delivery skill 的接口契约，必须包含：
- 来源 review run 路径。
- 已选择和明确不修的 finding IDs。
- 每个已选 finding 的证据位置、影响、修复方向和验收标准。
- 用户约束和禁止事项。
- 验证要求。

交接后，修复流程不得依赖聊天上下文猜测范围。

## No-Fix Closure

如果用户在 `02-review-findings.md` 后选择“不修”“先记录”“关闭”或等价表达：
- 不生成修复计划。
- 不改代码。
- 写 `07-review-summary.md`，交付类型标为“仅审查无修复”。
- 列出保留 findings、剩余风险和后续重新开启条件。

## Handoff 机器可读 YAML

`review-to-delivery-handoff.md` 必须同时有面向人的中文说明和机器可读 YAML。YAML 至少包含：
- `source_review_run`
- `selected_findings`
- `excluded_findings`
- `constraints`
- `verification`
- `forbidden_scope`

下游 skill 应优先用 YAML 校对范围，再读正文补充证据。

## 修复选择与计划的澄清循环

用户选择 findings 或确认修复方向时，AI 必须做一致性和可行性检查：
- 选中的 finding 是否依赖未选中的 finding。
- 修复方向是否与代码现状、接口契约、数据模型或项目约束冲突。
- 是否隐含架构调整、范围扩大或高风险操作。
- 是否存在更小、更稳妥的修复路径。

发现问题时，不进入实现；更新 `03-review-fix-selection.md` 或 `04-review-fix-plan.md`，提出选项，再请求确认。
